# ADR-0003 — Recovery of dolt 2.0.8 wisp-table corruption (Day-39, 2026-05-30)

Status: **OPEN — drop+re-migrate ATTEMPTED and reverted (nonlocal-federation wall); restored to known state. Needs beads-federation expertise or upstream fix.**
Date: 2026-05-30 (~11:40 PT)
Related: [[mc-jhsp8y]] soak (paused), study/notes/2026-05-30-day39-dolt-wisp-regression.md

## Context
Day-38 `gc upgrade` (HEAD-fad5d3f → HEAD-8d6d6bb, 2026-05-29 05:21 PT) pulled in **dolt 2.0.8**
(brew gave 2.0.8; gascity source pins 2.0.7). At **13:16:43 PT** (= 20:16 UTC) the new
beads layer ran a wisp schema migration that **created the `wisps` / `wisp_*` tables as
"nonlocal" (federated) tables** and populated them. During that write, dolt 2.0.8 mangled
the longtext adaptive-value encoding: every decode of `hq.wisps` longtext now panics
`invalid hash length: 19` (`AdaptiveValue.convertToTextStorage` → `hash.New`).

Consequences observed:
- Supervisor can't boot (snapshot load runs a wisp search → panic). "Not running" since 13:16.
- Beads **write** path blocked (`bd create` fails on the `0006_add_wisp_is_blocked` migration).
- mc-jhsp8y Day-39 re-soak could not fire. Soak paused.

## Decision (this ADR)
1. **Engine downgraded 2.0.8 → 2.0.4** (`/usr/local/bin/dolt` symlink → `Cellar/dolt/2.0.4`).
   Restores the pre-bug engine and stops further corruption. NECESSARY but NOT sufficient —
   2.0.4 also can't read the already-mangled bytes (proves a 2.0.8 **write-side** corruption,
   not a read regression).
2. **Full filesystem backup taken** (server stopped, clean SIGTERM):
   `/Users/rfvitis/my-city/.beads/dolt.backup-20260530T1129-pre-reset` (5.7G, verified). This
   is the only undo. NO migration backup exists (`gc dolt rollback` → "No backups found").
3. **Did NOT force a dolt reset** (guardrail #4). Read-only analysis shows a clean reset point
   does not exist:
   - `wisps` table is **absent** AS OF the last pre-migration commit `21qi4je2` (13:15:42 PT) —
     the migration created it; there is no earlier "readable wisps" state.
   - wisps were **not** stored as ephemeral `issues` pre-migration either (0 `mc-wisp-%` in
     `issues` AS OF `21qi4je2`). Source of the migrated rows is the nonlocal/federation path.
   - Loss bound: only **3** wisps written at/after the migration; the other ~7482 are
     migration-inserted rows (original `updated_at` preserved). Wisps are ephemeral.
   - A reset to pre-migration removes the wisps table and reverts schema → beads re-runs the
     same nonlocal-table migration that corrupted it. Reset can't land cleanly. Stopped.

System left as-found (no destructive change): dolt server back up on 2.0.4, real beads
readable (`gc bd show mc-jhsp8y` OK), supervisor still down (expected), backup intact.

## Options for the corrupt wisp tables (decision pending)
- **A (recommended): drop `wisps` + `wisp_*`, let beads re-run the wisp migration under 2.0.4.**
  Targeted to the ephemeral tables; real beads untouched; testable with the backup as undo.
  Hinge: does the migration write cleanly under 2.0.4 (no 2.0.8 bug)? If it re-corrupts,
  restore backup and escalate upstream.
- B: `dolt reset --hard 21qi4je2` then re-migrate — heavier, also discards uncommitted
  working-set changes to real tables (issues/comments/labels). Riskier for real data.
- C: wait for an upstream-fixed dolt / beads migration, then re-run. Safest; blocks recovery.

## Attempt log — drop + re-migrate (2026-05-30 ~12:00–12:30 PT), REVERTED
User authorized "drop wisp tables + re-migrate" (backup verified first). Outcome: **not viable
at the dolt level.**
1. Dropped `wisps` + 5 `wisp_*` tables (FK off). Verified gone.
2. `bd migrate schema` FAILED: `0007_recompute_wisp_is_blocked.up.sql: table not found: wisps`.
   Beads marks table-CREATION migrations applied/ignored, so dropping doesn't trigger
   recreation — and forward migrations then fail on the missing table, blocking ALL `bd`
   commands (DB-open runs the migration check).
3. Tried to restore tables from HEAD (`DOLT_CHECKOUT`): "tablespec did not match any table".
   Root insight: at dolt HEAD (`l2iocvjg` "disable nonlocal tables for fk migrations") there
   are **0 wisp tables**, `dolt_status` empty, drop created no commit. **The wisp tables are
   beads "nonlocal"/federated objects materialized at runtime, NOT committed dolt tables.**
   dolt-level drop/checkout/reset can't manage them.
4. Server restart did NOT re-materialize them. Live DB left without wisp tables (bd fully
   broken) → **restored the 5.7G backup**. Back to known state: 6 wisp tables, 7485 rows,
   real beads readable, engine 2.0.4. Broken state preserved at
   `.beads/dolt.BROKEN-postsurgery-20260530` for forensics.

CONCLUSION: recovery requires beads' OWN federation/migration tooling (or a fixed
dolt/beads), not dolt surgery. The corruption is entangled in the nonlocal-table federation
that the broken migration created. Out of safe scope without beads-internal expertise.
